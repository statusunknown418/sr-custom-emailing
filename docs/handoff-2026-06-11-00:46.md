## Fix applied this session (the one runtime bug found)

- **Clay 401 "Missing auth token".** `clay.ts` was sending `Authorization: <token>`
  (bare token, no `Bearer`). Clay accepts only `x-clay-webhook-auth: <TOKEN>`
  (raw) **or** `Authorization: Bearer <TOKEN>`. Changed `sendCommentersToClay` to
  send the dedicated header **`x-clay-webhook-auth: <CLAY_ENRICHER_AUTH_TOKEN>`**.
  Docs updated (`impl-plan.md`, `project_testing.md`). The env var was never the
  problem — the request reached Clay and was rejected on header format.
- To roll out a code change to a running task: `bun run deploy:background`, then
  **Replay** the failed run (its payload keeps `datasetId`, so it re-fetches the
  dataset and re-POSTs — no need to re-harvest).

## Clay-side config notes (return path: enriched lead -> our generate endpoint)

- Clay HTTP API runs **per row**; each call posts ONE lead.
- POST to (under the `/api-reference` prefix):
  - Flow A: `/api-reference/our-linkedin-comment-tracking/generate`
  - Flow B: `/api-reference/someone-else-post-scraping/generate`
- Header `x-internal-secret: <INTERNAL_API_SECRET>` (same value as the Worker;
  store it in the Clay "HTTP API (Headers) account", not plaintext on the column).
- Body: `{ "leads": [ { ...keys exactly matching clayLeadSchema... } ] }`. Only
  `originalPostUrl` is required; other fields normalize to "".
- **Routing by `flow`:** two HTTP API columns, each with a Clay run-condition
  (`flow == comment_tracking` / `flow == someone_else`) pointing at its endpoint.
- **Array mapping:** earlier the user saw only the first `leads[]` element map in
  Clay; they resolved it Clay-side (array fan-out). **No server change needed** —
  the batch payload `{ flow, originalPostUrl, leads: [...] }` stays as-is.
- **Field usage reality** (checked in both generate tasks): downstream only reads
  Flow A -> `name`, `personalLinkedinUrl`, `originalPostUrl`; Flow B -> `email`,
  `name`, `companyName`, `originalPostUrl`. All other schema fields
  (`companyUrl`, `companyLinkedin`, `companyEmployees`, `companyIndustry`,
  `companyDescription`, `country`, `staffinClassification`, `originalComment`) are
  validated + stored but currently unused. `companyUrl` = website,
  `companyLinkedin` = LinkedIn page — do not cross-wire them.

## Gotchas / invariants to remember

- **Ordering:** the post copy scrape must finish before Clay's enriched leads
  return, or the generate task FAILS loudly with
  `Cannot generate DMs/emails; posts not scraped yet: <url>` and writes nothing.
- **Reachability:** Apify's completion webhook calls the Worker directly, so
  `INTERNAL_API_URL` must be publicly reachable — use the deployed topology; pure
  localhost dev can't receive the callback.
- Both new tasks are `maxAttempts: 1` (non-idempotent: an Apify run / a Clay POST).
  Re-trigger or Replay manually on failure.
